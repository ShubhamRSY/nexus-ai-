Nexus — Production Journal (Oracle Free VM)

A “from zero to production” walkthrough of what we built, why the choices were made, and how to operate it safely.

What we built (high level)

Nexus is an omnichannel AI agent console and backend with three experiences:

  • Chat: end-user chat with tools + RAG, sessions, and streaming support
  • Copilot: agent-assist drafting/summarization for a human support agent
  • Voice: voice simulation and telephony integrations (Twilio, etc.)

All channels share:

  • One orchestrator (routing, tool-calling, and response generation)
  • One knowledge base (RAG / Chroma persistence)
  • One auth + multi-tenant model (JWT; tenant-scoped data)
  • One operational surface (health, observability, audit log, tasks)

Production goals

  • TLS/HTTPS at the edge
  • Authentication required for API access
  • No demo data in production
  • Locked registration (bootstrap once; then closed)
  • Backups + restore that work on the chosen persistence model
  • Process management that survives reboot (systemd)
  • Security headers to harden the UI
  • Rate limiting to resist abuse
  • Audit logging for sensitive admin actions

Deployment context (why we chose this path)

We deployed on Oracle’s Always Free VM, initially on a very small box. Docker was a poor fit for that size, so we used:

Bare-metal Python
Run the API with uvicorn
SQLite
Free + simple persistence (baseline)
Chroma
Vector store persisted under data/chroma
Caddy
HTTPS + security headers at the edge

This is not the “maximum scale” architecture, but it is a pragmatic, zero-cost production baseline.

Security baseline (what changed)

  1. Require JWT auth in production
    • AUTH_REQUIRED=true gates endpoints under /api/v1/* via Authorization: Bearer <JWT>
    • UI includes an auth gate and stores the token client-side
  2. Close registration after bootstrap
    • First admin can create the workspace
    • ALLOW_REGISTRATION=false closes /auth/register for public self-signup
  3. Disable demo behavior
    • DEMO_MODE=false prevents demo seeding
    • APP_ENV=production hides docs and disables demo reset routes
  4. Integrations are admin-only
    • Integrations write endpoints require admin JWT
    • Sensitive changes write audit logs
    • Integrations UI is hidden for non-admin users
  5. Rate limiting
    • RateLimitMiddleware enforces per-endpoint-group limits
    • Uses Redis if available; falls back to in-memory counters

Operations baseline

systemd service (survive reboot)

  • Installed using scripts/setup-nexus-systemd-oraclelinux.sh
  • Handles Oracle Linux SELinux constraints (venv executable context)

Daily backups (SQLite + Chroma)

  • scripts/backup.sh creates SQLite snapshot, Chroma archive, and config archive
  • scripts/setup-backup-timer.sh installs nexus-backup.timer (daily + retention)
  • Restore steps: docs/ops-backup-restore.md

Admin workflow

After bootstrap, admins create additional users via:

  • POST /api/v1/admin/users (admin-only)

This avoids re-opening public registration while still enabling team onboarding.

How to verify production is healthy

  • From the server: curl -fsS http://127.0.0.1:8001/api/v1/health
  • From the internet: curl -fsS https://<domain>/api/v1/health
  • Auth check: POST /api/v1/chat without JWT should return 401

Known limitations (free tier realities)

  • A micro VM + SQLite is not ideal for high concurrency.
  • For real production scale while staying free, prefer Oracle Ampere A1 Flex (Always Free) and run:
    • Postgres + Redis (Compose) + Nexus + Caddy

Next steps (industry-grade hardening)

  • External uptime monitoring + alerting (Sentry/OTEL supported)
  • Separate “admin console” from customer-facing UI
  • WAF / bot protection (Cloudflare)
  • Postgres + Redis for consistency and performance
